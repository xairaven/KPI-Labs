	\subsection*{Завдання}
	
	Створити закритий ключ та запит на сертифікат для доменного імені, яке відповідає варіанту завдання. Підписати запит на сертифікат, використовуючи сертифікат і закритий ключ центру сертифікації (CA), створеного в лабораторній роботі №2.
	
	Встановити та налаштувати файловий сервер, який реалізує протоколи FTP, FTPS та SFTP та відповідає наступним вимогам:
	
	\begin{itemize}
		\item анонімний доступ (користувач ftp або anonymous);
		
		\item в домашньому каталозі анонімного користувача створені 2 каталоги: \texttt{pub} та \texttt{incoming}, каталог \texttt{pub} має права тільки читання, каталог \texttt{incoming} має права на читання та запис;
		
		\item створені користувачі з іменами, доступом та правами, які відповідають варіанту завдання;
		
		\item для створених користувачів забезпечується доступ по протоколам, які відповідають варіанту завдання.
	\end{itemize}
	
	Виконати аналіз протокольного обміну між клієнтом та сервером під час автентифікації та пересилки файлів.
	
	Рекомендується використовувати наступне програмне забезпечення:
	
	\begin{itemize}
		\item \textbf{FTP-сервер}: vsftpd + openssh або proftpd;
		
		\item \textbf{FTP-клієнт}: lftp.
	\end{itemize}
	
	Для перевірки роботи файлового серверу та аналізу протокольного обміну рекомендується використовувати утиліти: \texttt{ftp}, \texttt{telnet}, \texttt{netcat}, \texttt{openssl}, \texttt{lftp}.
	
	Додати запис типу A для доменного імені файлового серверу в DNS. Додати сертифікат власного СА у список довірених на вузлі, де запускається FTP-клієнт.

	\section*{Хід роботи.}
	
	Спочатку, додамо потрібне налаштування в конфігураційний файл BIND9. Додаємо в файл \texttt{/etc/bind/db.cat.com} рядок: \sideat{[VM 1]}
	
	\begin{minted}{text}
		ftp	IN	A	192.168.1.2
	\end{minted}
	
	Також, треба закоментувати канонічне ім'я, бо воно посилається не на ту адресу.
	
	\begin{minted}{text}
		; ftp	IN	CNAME	lion
	\end{minted}
	
	Звичайно ж, додаємо запис у зворотню зону (\texttt{/etc/bind/db.1.168.192}):
	
	\begin{minted}{text}
		2	IN	PTR	ftp.cat.com.
	\end{minted}

		# Підписуємо запит сертифікатом
		user1@mail:~/Lab3$ sudo openssl x509 -req -in ftp.cat.com.csr -CA /usr/local/share/ca-certificates/ca.crt -CAkey /root/ca/ca.key -CAcreateserial -out ftp.cat.com.crt -days 365

	Користувачі \texttt{lion} та \texttt{leopard} мають доступ тільки по SFTP, тому для них файли в \texttt{vsftpd} не створюємо. Це частина сервісу SSH, а не \texttt{vsftpd}. Налаштування для них прописуються в \texttt{/etc/ssh/sshd\_config}, і вони ігнорують усе, що написано в папці \texttt{user\_conf} для FTP.
	
	Додаємо в файл \texttt{/etc/ssh/sshd\_config} таку опцію:
	
	\begin{minted}{text}
		# override default of no subsystems
		#Subsystem	sftp	/usr/lib/openssh/sftp-server
		Subsystem	sftp	internal-sftp
	\end{minted}
	
	Також, треба додати налаштування відносно вказаних у завданні користувачів.
	
	\begin{minted}{text}
		# SFTP Settings
		
		# lynx (SFTP | Home Directory | RW)
		Match User lynx
			ForceCommand internal-sftp
			ChrootDirectory %h
			AllowTcpForwarding no
			X11Forwarding no
			PasswordAuthentication yes
		
		# lion (SFTP | Home Directory | RW)
		Match User lion
		ForceCommand internal-sftp
			ChrootDirectory %h
			AllowTcpForwarding no
			X11Forwarding no
			PasswordAuthentication yes
		
		# leopard (SFTP | Full Tree | RW)
		Match User leopard
			ForceCommand internal-sftp
			AllowTcpForwarding no
			X11Forwarding no
			PasswordAuthentication yes
	\end{minted}
	
	Важливий нюанс OpenSSH: щоб \texttt{ChrootDirectory \%h} працювало, домашні директорії \texttt{/home/lynx} та \texttt{/home/lion} повинні належати \texttt{root} і мати права \texttt{755}. Змінимо власника: \texttt{sudo chown root:root /home/lynx /home/lion}.

		user1@mail:~$ sudo chmod 755 /home/jaguar
		user1@mail:~$ sudo chmod 755 /home/lion
	\end{minted}

	\subsection*{Висновок}
	
	Під час виконання лабораторної роботи було успішно налаштовано файловий сервер, що підтримує протоколи передачі даних FTP, FTPS та SFTP. Було створено та підписано сертифікат для захищеного з'єднання, а також налаштовано систему доменних імен для зручного доступу до сервера. За допомогою служби vsftpd реалізовано детальний розподіл прав доступу: налаштовано анонімний вхід із розділенням прав на читання та запис, а також створено п'ятьох локальних користувачів із різними рівнями доступу, включаючи ізоляцію в домашніх каталогах за допомогою механізму chroot. Для забезпечення роботи SFTP було інтегровано підсистему internal-sftp служби OpenSSH та вирішено архітектурні конфлікти прав доступу ізольованих каталогів шляхом створення вкладених директорій. Додатково було проведено глибокий аналіз мережевого трафіку за допомогою утиліт telnet, netcat, openssl та Wireshark. Практично доведено вразливість відкритого протоколу FTP, який передає автентифікаційні дані в незашифрованому вигляді, та наочно продемонстровано надійність протоколів FTPS і SFTP, що повністю інкапсулюють керуючі команди та дані в захищений криптографічний тунель.
	
	\section*{Контрольні запитання.}
	
	\begin{enumerate}
		\item \textbf{Протокол FTP призначення та особливості роботи.} \\ Протокол FTP (File Transfer Protocol) призначений для передачі файлів у комп'ютерних мережах на базі TCP/IP. Його головною архітектурною особливістю є використання двох окремих каналів з'єднання: керуючого каналу для передачі команд та відповідей, який зазвичай працює на порту 21, та каналу даних для безпосередньої передачі файлів. Класичний FTP не використовує шифрування, тому всі дані, включаючи логіни та паролі, передаються у відкритому текстовому вигляді.
		
		\item \textbf{Пасивний та активний режими роботи FTP-сервера.} \\ В активному режимі клієнт відкриває випадковий порт для отримання даних і повідомляє його серверу через команду PORT, після чого сервер ініціює з'єднання зі свого 20-го порту на вказаний порт клієнта. Цей режим часто блокується клієнтськими фаєрволами. У пасивному режимі клієнт надсилає команду PASV, у відповідь на яку сервер відкриває випадковий непривілейований порт і повідомляє його клієнту. Після цього клієнт сам ініціює з'єднання з цим портом сервера для передачі даних, що є більш безпечним та сумісним із сучасними NAT-маршрутизаторами.
		
		\item \textbf{Команди протоколу FTP.} \\ Команди FTP -- це текстові рядки, які клієнт надсилає серверу через керуючий канал для виконання певних дій. До основних команд належать USER та PASS для автентифікації користувача, LIST для отримання списку файлів і каталогів, CWD для зміни поточної робочої директорії, PWD для відображення поточного шляху. Для передачі файлів використовуються команди RETR для завантаження файлу з сервера на клієнт та STOR для збереження файлу на сервері. Для керування режимами з'єднання застосовуються команди PORT та PASV.
		
		\item \textbf{Протокол FTPS (FTP зверху SSL/TLS).} \\ Протокол FTPS є розширенням класичного протоколу FTP, яке додає підтримку криптографічних протоколів SSL та TLS для захисту переданих даних. Існує два методи ініціалізації захищеного з'єднання: неявний (Implicit), при якому з'єднання одразу встановлюється через захищений порт 990, та явний (Explicit), при якому клієнт підключається до стандартного порту 21 і надсилає команду AUTH TLS для переходу в зашифрований режим. FTPS дозволяє шифрувати як керуючий канал, так і канал даних, забезпечуючи конфіденційність автентифікаційних даних та цілісність файлів, що передаються.
		
		\item \textbf{Протокол SFTP.} \\ Протокол SFTP (SSH File Transfer Protocol) не має нічого спільного з класичним FTP, окрім схожої назви та призначення. Він є незалежним протоколом, який працює поверх безпечного каналу зв'язку, найчастіше як підсистема протоколу SSH на порту 22. SFTP з самого початку проектувався як безпечний протокол: він забезпечує повне шифрування всіх даних та команд, використовує єдине з'єднання для керування і передачі інформації, а також надає ширші можливості для управління файловою системою, включаючи зміну прав доступу та власників файлів.
	\end{enumerate}
