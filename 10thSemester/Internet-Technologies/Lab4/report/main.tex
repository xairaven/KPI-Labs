	\subsection*{Мета роботи} 
	
	Отримати практичні навички роботи з мережною файловою системою NFS.
	
	\subsection*{Завдання}
	
	Встановити та налаштувати файловий сервер, який реалізує мережну файлову систему NFS та відповідає наступним вимогам:
	
	\begin{itemize}
		\item створені каталоги: \texttt{/nfs/public}, \texttt{/nfs/private}, \texttt{/nfs/incoming};
		
		\item каталог \texttt{/nfs/public} експортується в режимі тільки читання;
		
		\item каталог \texttt{/nfs/private} експортується в режимі читання та запису і тільки для вузла, на якому працює NFS-клієнт, запити від всіх користувачів на клієнті відображаються на заданого варіантом користувача на сервері, який має повні права в цьому каталозі;
		
		\item каталог \texttt{/nfs/incoming} експортується в режимі читання та запису для вузлів, які знаходяться в тій же мережі, що і NFS-клієнт, запити від користувача \texttt{root} на клієнті відображаються на заданого варіантом користувача на сервері, запити інших користувачів клієнта залишаються без змін на сервері, права на каталог встановлені відповідно варіанту завдання.
	\end{itemize}
	
	Виконати аналіз протокольного обміну між клієнтом та сервером під час монтування файлової системи та пересилки файлів.
	
	Рекомендується використовувати наступне програмне забезпечення:
	
	\begin{itemize}
		\item \textbf{NFS-сервер}: nfs-kernel-server;
		
		\item \textbf{NFS-клієнт}: nfs-common.
	\end{itemize}
	
	Для перевірки роботи файлового серверу та аналізу протокольного обміну рекомендується використовувати утиліти: \texttt{nfsstat}, \texttt{showmount}, \texttt{mount}, \texttt{tcpdump}.
	
	Завдання за варіантом:
	
	\begin{itemize}
		\item \textbf{Ім'я NFS-серверу}: \texttt{nfs.cat.com}.
		
		\item \textbf{Відображення користувачів}:
		
		\begin{itemize}
			\item \texttt{/nfs/private}: \texttt{tiger};
			
			\item \texttt{/nfs/incoming}: \texttt{lion}.
		\end{itemize}
		
		\item \textbf{Права на каталог \texttt{/nfs/incoming}}:
		
		\begin{itemize}
			\item RWX: \texttt{lynx};
			
			\item -: \texttt{leopard};
			
			\item RX: \texttt{jaguar};
			
			\item RWX: \texttt{lion}.
		\end{itemize}
	\end{itemize}
	
	\section*{Хід роботи.}
	
	Для початку, додамо відповідні записи в налаштування зон DNS.
	
	Файл \texttt{/etc/bind/db.cat.com}: \sideat{[VM 1]}
	
	\begin{minted}{text}
		nfs	IN	A	192.168.1.2
	\end{minted}
	
	Зворотня зона -- \texttt{/etc/bind/db.1.168.192}: \sideat{[VM 1]}
	
	\begin{minted}{text}
		2	IN	PTR	nfs.cat.com.
	\end{minted}
	
	Звичайно ж, інкрементуємо серійний номер в обох файлах. Після цього, перезавантажимо сервіс.
	
	\image[width=18cm]{10.7cm}{bind9-restart}
	
	Встановимо необхідні пакети.
	
	Сервер: \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ sudo apt install nfs-kernel-server acl
	\end{minted}
	
	Клієнт: \sideat{[VM 2]}
	
	\begin{minted}{bash}
		user2@mail:~$ sudo apt install nfs-common
	\end{minted}
	
	Створимо відповідні каталоги. \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ sudo mkdir -p /nfs/{public,private,incoming}
	\end{minted}
	
	Так як треба зробити відображення користувачів на директорії, нам треба дізнатися їх ідентифікатори. \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ id tiger
		uid=1002(tiger) gid=1002(tiger) groups=1002(tiger)
		
		user1@mail:~$ id lion
		uid=1003(lion) gid=1003(lion) groups=1003(lion)
	\end{minted}
	
	\pagebreak
	
	Тепер роздамо локальні права на папки.
	
	Папка \texttt{public} має бути доступна всім тільки для читання. За замовчуванням при створенні вона має права 755 (читати та заходити можуть усі), тому тут нічого не міняємо. \sideat{[VM 1]}

	\begin{minted}{bash}
		user1@mail:~$ ls -la /nfs/public/
		total 8
		drwxr-xr-x 2 root root 4096 Apr 17 02:34 .
		drwxr-xr-x 5 root root 4096 Apr 17 02:34 ..
	\end{minted}
	
	Папка \texttt{private} -- за варіантом туди мають проектуватися всі запити від імені \texttt{tiger} з повними правами. Отже, робимо його повноправним власником цієї папки: \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ sudo chown tiger:tiger /nfs/private
		
		user1@mail:~$ sudo chmod 700 /nfs/private
		
		user1@mail:~$ sudo ls -la /nfs/private
		total 8
		drwx------ 2 tiger tiger 4096 Apr 17 02:34 .
		drwxr-xr-x 5 root  root  4096 Apr 17 02:34 ..
	\end{minted}
	
	Папка \texttt{incoming} -- тут запити від \texttt{root} мають ставати \texttt{lion}. Віддамо папку йому. \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ sudo chown lion:lion /nfs/incoming
		
		user1@mail:~$ sudo ls -la /nfs/incoming
		total 8
		drwxr-xr-x 2 lion lion 4096 Apr 17 02:34 .
		drwxr-xr-x 5 root root 4096 Apr 17 02:34 ..
	\end{minted}
	
	За допомогою POSIX Access Control Lists видамо права на директорію \texttt{/nfs/incoming}. \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ sudo setfacl -m u:lynx:rwx /nfs/incoming
		
		user1@mail:~$ sudo setfacl -m u:leopard:--- /nfs/incoming
		
		user1@mail:~$ sudo setfacl -m u:jaguar:r-x /nfs/incoming
		
		user1@mail:~$ sudo setfacl -m u:lion:rwx /nfs/incoming
	\end{minted}
	
	Перевіримо: \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ getfacl /nfs/incoming
		getfacl: Removing leading '/' from absolute path names
		# file: nfs/incoming
		# owner: lion
		# group: lion
		user::rwx
		user:lion:rwx
		user:lynx:rwx
		user:leopard:---
		user:jaguar:r-x
		group::r-x
		mask::rwx
		other::r-x
	\end{minted}
	
	Далі налаштуємо NFS. Повний зміст файлу \texttt{/etc/exports}: \sideat{[VM 1]}
	
	\begin{minted}{text}
		# 1. Public: Reading for all users
		/nfs/public  *(ro,sync,no_subtree_check)
		
		# 2. Private: export only for the client (192.168.1.3), RW.
		# All users (all_squash) mapped as tiger (uid=1002)
		/nfs/private  192.168.1.3/32(rw,sync,no_subtree_check,all_squash,anonuid=1002,anongid=1002)
		
		# 3. Incoming: export for whole subnet. RW.
		# Root (root_squash) mapped as lion (uid=1003)
		/nfs/incoming  192.168.1.0/24(rw,sync,no_subtree_check,root_squash,anonuid=1003,anongid=1003)
	\end{minted}
	
	Застосуємо зміни. \sideat{[VM 1]}
	
	\begin{minted}{bash}
		user1@mail:~$ sudo exportfs -arv
	\end{minted}

	\subsection*{Висновок}
	
	Під час виконання лабораторної роботи було успішно встановлено та налаштовано сервер мережевої файлової системи NFS. Було створено структуру каталогів та реалізовано гнучку політику розмежування прав доступу з використанням параметрів конфігураційного файлу \texttt{/etc/exports} (зокрема \texttt{ro}, \texttt{rw}, \texttt{root\_squash}, \texttt{all\_squash}). На практиці було успішно налаштовано трансляцію ідентифікаторів користувачів (UID/GID) між клієнтом та сервером (squashing) для забезпечення коректного доступу до приватних та вхідних каталогів. Додатково було застосовано класичні механізми UNIX-прав (через хитре налаштування власника, груп та специфічних дозволів. Було протестовано ручне монтування та автоматичне монтування через таблицю \texttt{/etc/fstab} із використанням параметра \texttt{\_netdev} для безпечного завантаження клієнта. В результаті аналізу мережевого трафіку за допомогою аналізатора Wireshark та утиліти \texttt{nfsstat} було досліджено механізм віддаленого виклику процедур (RPC), роботу служби MOUNT та процес безпосередньої передачі файлів протоколом NFS.
	
	\section*{Контрольні запитання.}
	
	\begin{enumerate}
		\item \textbf{Відповідність NFS моделі ISO OSI.} \\ Протокол NFS працює на прикладному рівні (Рівень 7) моделі OSI. Для своєї роботи він тісно взаємодіє з механізмом RPC (Remote Procedure Call), який функціонує на сеансовому рівні (Рівень 5), та стандартом представлення даних XDR (eXternal Data Representation) на рівні представлення (Рівень 6). На транспортному рівні (Рівень 4) класичний NFS використовував UDP, але сучасні версії (NFSv3/NFSv4) за замовчуванням використовують більш надійний протокол TCP. На мережевому рівні (Рівень 3) використовується протокол IP.
		
		\item \textbf{Архітектура NFS.} \\ NFS побудована за класичною клієнт-серверною архітектурою. Сервер експортує (робить доступними) локальні каталоги через мережу, а клієнт монтує їх у свою локальну файлову систему так, ніби це фізично підключені жорсткі диски. Прозорість доступу забезпечується компонентом VFS (Virtual File System) в ядрі Linux. Коли користувач або програма звертається до примонтованого каталогу, VFS перехоплює системні виклики і, розпізнавши мережевий шлях, передає їх NFS-клієнту. Той, у свою чергу, формує RPC-запити до NFS-сервера для виконання файлових операцій.
		
		\pagebreak
		
		\item \textbf{Призначення та функції компонент NFS: \texttt{mountd}, \texttt{nfsd}, \texttt{rpcbind} (\texttt{portmapper}).} \\ \begin{itemize}
			\item \textbf{\texttt{rpcbind} (або \texttt{portmapper}):} Служба, яка зіставляє номери програм RPC з відповідними мережевими портами (TCP/UDP). Оскільки служби NFS можуть запускатися на динамічних портах, клієнт спочатку звертається до \texttt{rpcbind} (порт 111), щоб дізнатися, на якому порту працює потрібна йому служба (наприклад, \texttt{mountd}).
			\item \textbf{\texttt{mountd}:} Служба монтування. Вона приймає запити від клієнтів на підключення, перевіряє права доступу згідно з конфігураційним файлом \texttt{/etc/exports} і, у разі дозволу, генерує та передає клієнту початковий файловий дескриптор (File Handle) для запитуваного каталогу.
			\item \textbf{\texttt{nfsd}:} Головний демон сервера (який працює на рівні ядра операційної системи). Він безпосередньо обробляє всі файлові операції (читання, запис, відкриття, закриття файлів) після того, як каталог був успішно змонтований клієнтом.
		\end{itemize}
		
		\item \textbf{Процедура монтування файлової системи.} \\ Процедура монтування (у класичній архітектурі NFSv3) складається з наступних етапів. Спочатку клієнт відправляє RPC-запит до \texttt{rpcbind} на сервері, щоб дізнатися порт служби \texttt{mountd}. Отримавши порт, клієнт відправляє запит протоколу MOUNT до \texttt{mountd} із зазначенням шляху до каталогу. Сервер перевіряє файл \texttt{/etc/exports}, щоб з'ясувати, чи має право цей клієнт (за його IP-адресою) підключати каталог. Якщо доступ дозволено, \texttt{mountd} повертає клієнту унікальний ідентифікатор -- \textit{File Handle} кореня експортованого каталогу. Після цього клієнт прив'язує цей дескриптор до локальної точки монтування, і всі подальші звернення до файлів ідуть безпосередньо до демона \texttt{nfsd} за допомогою отриманого дескриптора. \textit{(Примітка: в NFSv4 архітектуру спрощено -- протокол MOUNT інтегровано в основний протокол, і всі запити обробляються виключно через єдиний порт 2049)}.
	\end{enumerate}
