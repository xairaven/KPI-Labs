	\subsection*{Завдання}
	
	Створити закритий ключ та запит на сертифікат для доменного імені, яке відповідає варіанту завдання. Підписати запит на сертифікат, використовуючи сертифікат і закритий ключ центру сертифікації (CA), створеного в лабораторній роботі №2.
	
	Встановити та налаштувати Web-сервер, який реалізує протоколи HTTP та HTTPS, виконує функції балансування навантаження для двох Web-серверів та відповідає наступним вимогам:
	
	\begin{itemize}
		\item доменне ім’я сервера балансування навантаження відповідає варіанту завдання;
		
		\item сервер балансування навантаження слухає порти для протоколів HTTP та HTTPS, номери портів відповідають варіанту завдання;
		
		\item доступ до сервера балансування навантаження мають тільки користувачі, що пройшли автентифікацію, імена яких відповідають варіанту завдання;
		
		\item метод балансування навантаження відповідає варіанту завдання;
	\end{itemize}
	
	Встановити та налаштувати 2 Web-сервери, які реалізують протокол HTTP та виконують роль серверів обробки запитів клієнтів.
	
	Виконати аналіз протокольного обміну між клієнтом та сервером балансування навантаження під час виконання запитів по протоколам HTTP та HTTPS.
	
	Рекомендується використовувати наступне програмне забезпечення:
	
	\begin{itemize}
		\item \textbf{Web-сервер}: \texttt{nginx} або \texttt{apache};
		
		\item \textbf{Web-клієнт}: \texttt{curl}.
	\end{itemize}
	
	Для перевірки роботи Web-серверу та аналізу протокольного обміну рекомендується використовувати утиліти: \texttt{telnet}, \texttt{netcat}, \texttt{openssl}, \texttt{curl}.
	
	Додати запис типу \texttt{A} для доменного імені Web-серверу в DNS. Додати сертифікат власного СА у список довірених на вузлі, де запускається Web-клієнт.
	
	\textbf{Завдання за варіантом:}
	
	\begin{itemize}
		\item \textbf{Ім'я Web-серверу}: \texttt{www.cat.com}
		
		\item \textbf{Порт HTTP}: 8083
		
		\item \textbf{Порт HTTPS}: 8446
		
		\item \textbf{Користувачі}: \texttt{tiger}, \texttt{lion}, \texttt{lynx}
		
		\item \textbf{Метод балансування}: ip-hash
	\end{itemize}

	В цей файл (\texttt{/etc/nginx/sites-available/loadbalancer}) записуємо ці інструкції: \sideat{[VM 1]}
	
	\begin{minted}{text}
		# Group of servers
		upstream backend {
			hash $remote_addr;
			server 127.0.0.1:8001;
			server 192.168.1.3:80;
		}
		
		# Local Backend #1
		server {
			listen 127.0.0.1:8001;
			root /var/www/html;
			index index.html;
		}
		
		# Fronted on 8083
		server {
			listen 8083;
			server_name www.cat.com;
			
			auth_basic "Restricted Access";
			auth_basic_user_file /etc/nginx/.htpasswd;
			
			location / {
				proxy_pass http://backend;
			}
		}
		
		# SSL Frontend
		server {
			listen 8446 ssl;
			server_name www.cat.com;
			
			ssl_certificate /etc/ssl/certs/www.cat.com.crt;
			ssl_certificate_key /etc/ssl/private/www.cat.com.key;
			
			auth_basic "Restricted Access";
			auth_basic_user_file /etc/nginx/.htpasswd;
			
			location / {
				proxy_pass http://backend;
			}
		}
	\end{minted}
	
	В файлі \texttt{/etc/nginx/nginx.conf} змінимо значення змінної \texttt{worker\_processes} з \texttt{auto} на \texttt{1}, щоб спростити відлагодження та демонстрацію.

	\subsection*{Висновок}
	
	Під час виконання лабораторної роботи було успішно розгорнуто та налаштовано веб-сервер Nginx у ролі зворотного проксі-сервера (reverse proxy) та балансувальника навантаження. Було створено закритий ключ і запит на сертифікат для доменного імені \texttt{www.cat.com}, який успішно підписано власним центром сертифікації (CA). Цей сертифікат було інтегровано у сховище довірених сертифікатів на клієнтській машині, що дозволило встановити безпечне з'єднання по протоколу HTTPS.
	
	Було налаштовано два фронтенди (HTTP на порту 8083 та HTTPS на порту 8446) із застосуванням базової автентифікації користувачів за допомогою утиліти htpasswd, що обмежило доступ до ресурсів лише для авторизованих користувачів (\texttt{tiger}, \texttt{lion}, \texttt{lynx}).
	
	Маршрутизацію трафіку між двома бекенд-серверами було реалізовано з використанням методу хешування IP-адреси клієнта. За допомогою утиліти \texttt{curl} було практично доведено коректність роботи механізмів авторизації та балансування навантаження. Шляхом додавання додаткових IP-адрес на клієнтську машину (VM 3) та виконання запитів з різних інтерфейсів було успішно продемонстровано перенаправлення запитів на різні бекенд-сервери (VM 1 та VM 2). Аналіз протокольного обміну за допомогою openssl s\_client підтвердив надійність криптографічного рукостискання (TLS Handshake) та правильність налаштування SSL-сертифікатів.
	
	\section*{Контрольні запитання.}
	
	\begin{enumerate}
		\item \textbf{Протокол HTTP.} \\ HTTP (HyperText Transfer Protocol) -- це протокол прикладного рівня моделі OSI, призначений для передачі гіпертекстових документів та інших даних. Він працює за архітектурою клієнт-сервер і є протоколом без збереження стану (stateless), тобто кожен запит обробляється незалежно від попередніх. За замовчуванням використовує TCP-порт 80.
		
		\item \textbf{Покажчики ресурсів URI та URL.} \\ URI (Uniform Resource Identifier) -- це уніфікований ідентифікатор ресурсу, рядок символів, який однозначно ідентифікує ресурс у мережі. URL (Uniform Resource Locator) є підмножиною URI, яка не лише ідентифікує ресурс, але й вказує механізм доступу до нього (наприклад, схему \texttt{https://} або \texttt{ftp://}) та його місцезнаходження на сервері.
		
		\item \textbf{Структура повідомлень HTTP.} \\ HTTP-повідомлення (як запити, так і відповіді) складаються з трьох основних частин: 1) Стартовий рядок (Request Line у запитах із вказанням методу та URI, або Status Line у відповідях із кодом стану); 2) HTTP-заголовки (Headers), які передають метадані у форматі <<Ключ: Значення>>; 3) Порожній рядок, що сигналізує про завершення заголовків; 4) Тіло повідомлення (Message Body) -- опціональна частина, що містить безпосередньо дані (HTML-код, JSON, зображення тощо).
		
		\item \textbf{Методи HTTP.} \\ Методи вказують на дію, яку клієнт хоче виконати над ресурсом. Основні методи: \texttt{GET} (отримання даних), \texttt{POST} (відправка нових даних на сервер), \texttt{PUT} (повне оновлення існуючого ресурсу), \texttt{DELETE} (видалення), \texttt{HEAD} (отримання лише заголовків без тіла), \texttt{OPTIONS} (запит параметрів з'єднання та підтримуваних методів).
		
		\item \textbf{Сесії в протоколі HTTP.} \\ Оскільки HTTP є протоколом без стану, для реалізації логіки сесій (наприклад, авторизації користувача на сайті) використовуються додаткові механізми. Найчастіше це технологія \texttt{Cookies} (сервер відправляє ідентифікатор сесії клієнту, і клієнт додає його до кожного наступного запиту), або токени доступу (наприклад, JWT), які передаються в заголовку \texttt{Authorization}.
		
		\item \textbf{Кешування в протоколі HTTP.} \\ Механізм збереження копій відповідей сервера для зменшення навантаження на мережу та прискорення завантаження сторінок. Кешування керується спеціальними HTTP-заголовками, такими як \texttt{Cache-Control} (визначає правила кешування), \texttt{Expires} (час життя кешу), \texttt{ETag} (унікальний хеш версії файлу) та \texttt{Last-Modified}.
		
		\item \textbf{Постійне з’єднання в протоколі HTTP.} \\ Механізм HTTP Keep-Alive. Починаючи з версії HTTP/1.1, TCP-з'єднання між клієнтом та сервером не закривається відразу після отримання однієї відповіді. Воно залишається відкритим для передачі кількох наступних HTTP-запитів і відповідей, що суттєво зменшує затримки (latency), оскільки усуває необхідність повторного виконання TCP-рукостискання (3-way handshake).
	\end{enumerate}
