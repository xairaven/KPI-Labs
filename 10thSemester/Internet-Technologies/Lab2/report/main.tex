	\subsection*{Завдання}
	Створити центр сертифікації (CA) \texttt{issuer} і \texttt{subject} якого відповідають варіанту завдання. Сертифікати для поштових серверів мають підписуватись сертифікатом та ключем створеного СА. Сертифікат створеного СА необхідно додати в список довірених на всіх вузлах, які використовуються в роботі.
	
	Встановити та налаштувати 2 поштових сервери (mail/message transfer agent -- MTA) для пересилки повідомлень електронної пошти за допомогою протоколу SMTP, які відповідають наступним вимогам:
	
	\begin{itemize}
		\item кожний поштовий сервер є кінцевим отримувачем повідомлень для домену, ім'я якого відповідає варіанту завдання;
		
		\item доменне ім'я поштового серверу має відповідати формату: \texttt{mail.<ім'я домену>};
		
		\item в кожному домені створені поштові скриньки з іменами, які відповідають варіанту завдання;
		
		\item обов'язкова автентифікація користувачів при відправленні повідомлень на зовнішні MTA;
		
		\item підтримка TLS для передачі повідомлень;
		
		\item вихідні повідомлення містять підпис DKIM;
		
		\item перевірка вхідних повідомлень на наявність та коректність підпису DKIM, повідомлення з некоректним підписом відкидаються;
		
		\item перевірка домену відправника повідомлення та обробка повідомлення у відповідності з SPF, яка відповідає варіанту завдання;
		
		\item перевірка повідомлень на наявність шкідливого програмного забезпечення;
		
		\item контекстний аналіз повідомлень для виконання спам-фільтрації.
	\end{itemize}
	
	Для кожного сервера додати функціонал доставлення повідомлень засобами протоколів POP3 та IMAP з підтримкою TLS.
	
	Виконати аналіз протокольного обміну між вузлами під час відправлення, пересилки та доставлення поштових повідомлень. Проаналізувати заголовки листів.
	
	Рекомендується використовувати наступне програмне забезпечення:
	
	\begin{itemize}
		\item MTA: exim або postfix;
		
		\item доставлення повідомлень засобами POP3 та IMAP: dovecot;
		
		\item антивірусна фільтрація: ClamAV;
		
		\item спам-фільтрація: SpamAssassin.
	\end{itemize}
	
	Кожний поштовий сервер рекомендується встановлювати та налаштовувати на окремій віртуальній машині. Для перевірки роботи модулів поштового сервера та аналізу протокольного обміну рекомендується використовувати утиліти: telnet, openssl, swaks.
	
	Додати необхідні записи типів: MX та TXT для забезпечення функціонування поштових серверів згідно з варіантом завдання.
	
	\begin{itemize}
		\item \textbf{Issuer та Subject CA}: 
		
		\begin{itemize}
			\item \texttt{C=UA}
			
			\item \texttt{ST=Ukraine}
			
			\item \texttt{L=Kyiv}
			
			\item \texttt{O=KPI}
			
			\item \texttt{OU=OT}
			
			\item \texttt{cat CA}
		\end{itemize}
		
		\item \textbf{Домен 1}: \texttt{cat.com}
		
		\item \textbf{Домен 2}: \texttt{zone03.net}
		
		\item \textbf{Поштові скриньки}: \texttt{tiger@cat.com}, \texttt{lion@cat.com}, \texttt{lynx@cat.com}, \texttt{leopard@zone03.net}, \texttt{jaguar@zone03.net}
		
		\item \textbf{Політика SPF}: \texttt{v=spf1 a mx -all}
	\end{itemize}

	\subsection*{Налаштування центру сертифікації}
	
	Робота була розпочата зі створення локального центру сертифікації, тобто довіреного вузлу, який підтверджує справжність криптографічних ключів за допомогою цифрових сертифікатів. Спочатку необхідно створити кореневий сертифікат центру сертифікації та його приватний ключ. Для цього на першій віртуальній машині була запущена команда генерації за допомогою утиліти OpenSSL. Вона створить закритий ключ та самопідписаний сертифікат терміном на десять років. Аргументи зі завдання були передані через параметр \texttt{subj}. 
	
	\begin{minted}{bash}
		# Generate Root CA private key and self-signed certificate
		openssl req -x509 -newkey rsa:4096 -days 3650 -nodes -keyout ca.key -out ca.crt -subj "/C=UA/ST=Ukraine/L=Kyiv/O=KPI/OU=OT/CN=cat CA"
	\end{minted}
	
	Далі потрібно створити сертифікати для самих поштових серверів та підписати їх створеним CA. Цю процедуру потрібно виконати для обох доменів. Почнемо з домену \texttt{cat.com}. Спочатку генерується закритий ключ для поштового сервера. Потім створюється запит на підпис сертифіката (Certificate Signing Request або CSR), де вказується ім'я поштового сервера \texttt{mail.cat.com}. Після цього цей запит підписується ключем кореневого центру сертифікації.
	
	\begin{minted}{bash}
		# Generate private key for mail.cat.com
		openssl genrsa -out mail.cat.com.key 2048
		
		# Create Certificate Signing Request (CSR) for mail.cat.com
		openssl req -new -key mail.cat.com.key -out mail.cat.com.csr -subj "/C=UA/ST=Ukraine/L=Kyiv/O=KPI/OU=OT/CN=mail.cat.com"
		
		# Sign the CSR with Root CA to generate the certificate
		openssl x509 -req -in mail.cat.com.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out mail.cat.com.crt -days 365
	\end{minted}
	
	Маємо результат:
	
	\begin{minted}{text}
		Certificate request self-signature ok
		subject=C=UA, ST=Ukraine, L=Kyiv, O=KPI, OU=OT, CN=mail.cat.com
	\end{minted}
	
	Аналогічну процедуру необхідно провести для другого поштового сервера у домені \texttt{zone03.net}. Змінюється лише доменне ім'я вузла (CN) на \texttt{mail.zone03.net}.
	
	\begin{minted}{bash}
		# Generate private key for mail.zone03.net
		openssl genrsa -out mail.zone03.net.key 2048
		
		# Create Certificate Signing Request (CSR) for mail.zone03.net
		openssl req -new -key mail.zone03.net.key -out mail.zone03.net.csr -subj "/C=UA/ST=Ukraine/L=Kyiv/O=KPI/OU=OT/CN=mail.zone03.net"
		
		# Sign the CSR with Root CA to generate the certificate
		openssl x509 -req -in mail.zone03.net.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out mail.zone03.net.crt -days 365
	\end{minted}
	
	Далі треба розмістити файли по відповідним машинам. Маємо таке розташування:
	
	\begin{minted}{text}
		user1@vm1:~/CA$ ls
		ca.crt  ca.key  ca.srl  mail.cat.com.crt  mail.cat.com.key
		
		user2@vm2:~/CA$ ls
		ca.crt  mail.zone03.net.crt  mail.zone03.net.key
	\end{minted}
	
	У Linux-системах, таких як Debian чи Ubuntu, існують загальноприйняті стандарти для зберігання криптографічних матеріалів. Дотримання цих правил не лише підвищує безпеку, але й значно спрощує подальше налаштування служб, оскільки багато з них за замовчуванням шукають ключі саме в цих місцях.
	
	Публічні сертифікати поштових серверів, а саме файли \texttt{mail.cat.com.crt} та \texttt{mail.zone03.net.crt}, найкраще покласти до директорії \texttt{/etc/ssl/certs/} на відповідних віртуальних машинах. Ця тека призначена саме для публічних даних. Сертифікати не є таємницею, тому їх можуть читати всі служби, яким потрібно забезпечити TLS-з'єднання.
	
	Приватні ключі серверів, тобто файли \texttt{mail.cat.com.key} та \texttt{mail.zone03.net.key}, є секретною інформацією. Їхнє стандартне місце -- директорія \texttt{/etc/ssl/private/} на відповідних машинах. Ця папка має спеціальні права доступу, які обмежують читання для звичайних користувачів. Після переміщення туди приватного ключа, обов'язково треба встановити на нього максимально суворі права. Це робиться для того, щоб ключ міг прочитати лише суперкористувач \texttt{root}.
	
	Що стосується файлів самого Центру Сертифікації, зокрема вашого головного приватного ключа \texttt{ca.key} та файлу \texttt{ca.srl}, їх взагалі не потрібно тримати поруч із робочими файлами пошти. Цей ключ є коренем довіри вашої інфраструктури. Найкраще створити окрему приховану теку безпосередньо в домашній папці адміністратора на першій машині, наприклад \texttt{/root/ca/}, і зберігати ці файли там. Поштові сервери ніколи не звертатимуться до цього ключа напряму під час своєї роботи.
	
	Відповідно, маємо такий порядок виконання команд на першій віртуальній машині:
	
	\begin{minted}{bash}
		user1@vm1:~/CA$ ls
		ca.crt  ca.key  ca.srl  mail.cat.com.crt  mail.cat.com.key
		user1@vm1:~/CA$ sudo mv mail.cat.com.key /etc/ssl/private/
		user1@vm1:~/CA$ sudo mv mail.cat.com.crt /etc/ssl/certs/
		user1@vm1:~/CA$ sudo chmod 600 /etc/ssl/private/mail.cat.com.key
		user1@vm1:~/CA$ sudo mkdir /root/ca/
		user1@vm1:~/CA$ ls
		ca.crt  ca.key  ca.srl
		user1@vm1:~/CA$ sudo mv ca.key  ca.srl /root/ca/
		user1@vm1:~/CA$ sudo mv ca.srl /root/ca/
		user1@vm1:~/CA$ sudo ls /root/ca/
		ca.key  ca.srl
	\end{minted}
	
	На другій:
	
	\begin{minted}{bash}
		user2@vm2:~/CA$ sudo mv mail.zone03.net.key /etc/ssl/private/
		user2@vm2:~/CA$ sudo mv mail.zone03.net.crt /etc/ssl/certs/
		user2@vm2:~/CA$ sudo chmod 600 /etc/ssl/private/mail.zone03.net.key
	\end{minted}
	
	На обох машинах, і на першій, і на другій, файл публічного сертифіката \texttt{ca.crt} потрібно розмістити в абсолютно однаковому місці, щоб операційні системи почали довіряти центру сертифікації.
	
	Цільова директорія для цього файлу в системах на базі Debian або Ubuntu -- це \texttt{/usr/local/share/ca-certificates/}. Це стандартне розташування для локально доданих кореневих сертифікатів. Дуже важливо, щоб файл зберігав розширення \texttt{.crt}, оскільки системні утиліти шукають саме такі файли під час оновлення сховища.
	
	Після копіювання, необхідно буде виконати команду оновлення. Ця команда змусить систему просканувати директорію, знайти новий сертифікат і офіційно додати його до загального системного списку довірених вузлів. Отримуємо однаковий результат на обох машинах.
	
	\begin{minted}{bash}
		user1@vm1:~/CA$ sudo mv ca.crt /usr/local/share/ca-certificates/
		user1@vm1:~/CA$ sudo update-ca-certificates
		Updating certificates in /etc/ssl/certs...
		rehash: warning: skipping ca-certificates.crt, it does not contain exactly one certificate or CRL
		1 added, 0 removed; done.
		Running hooks in /etc/ca-certificates/update.d...
		done.
	\end{minted}
	
	Щойно друга команда завершить роботу, у терміналі має з'явитися короткий звіт про те, що було додано один новий сертифікат. Це є підтвердженням того, що обидві машини відтепер розпізнаватимуть і довірятимуть сертифікатам поштових серверів. Це забезпечить коректну роботу TLS-з'єднань між ними без помилок перевірки автентичності.

	Змінюємо налаштування BIND9. Параметр \texttt{directory} задає базову робочу папку сервера для збереження кешу та файлів підлеглих зон. Опція \texttt{reuseport} вимикає зайвий для лабораторної механізм розподілу мережевого навантаження між ядрами процесора. Налаштування \texttt{listen-on} змушує сервер приймати запити виключно на вказаних адресах, що є базовим правилом безпеки. Нарешті, \texttt{listen-on-v6} повністю відключає протокол IPv6, оскільки у мережі використовується лише IPv4.
	
	Перша машина (файл -- \texttt{/etc/bind/named.conf.options}):
	
	\begin{minted}{text}
		options {
			directory "/var/cache/bind";
			reuseport no;
			listen-on port 53 { 192.168.1.2; 127.0.0.1; };
			listen-on-v6 { none; };
		};
	\end{minted}
	
	Дві зони вже були налаштовані в першій лабораторній роботі. Але, для другої роботи треба внести правки. Спочатку додаємо запис MX який посилається на машину <<mail>>, яка і є першою віртуальною. Треба додати два рядки в файл \texttt{/etc/bind/db.cat.com}:
	
	\begin{minted}{text}
		@	IN	MX	10 mail.cat.com.
		mail	IN	A	192.168.1.2
	\end{minted}
	
	Такі ж зміни і для \texttt{/etc/bind/db.zone03.net}, але посилаємося вже на другу віртуальну машину:
	
	\begin{minted}{text}
		@	IN	MX	10 mail.zone03.net.
		mail	IN	A	192.168.1.3
	\end{minted}
	
	Не забуваємо про зворотню зону (\texttt{/etc/bind/db.1.168.192}).
	
	\begin{minted}{text}
		; MAIL RECORDS
		2       IN      PTR     mail.cat.com.
		3       IN      PTR     mail.zone03.net.
	\end{minted}

	\subsection*{ClamAV} 
	
	Для початку, встановимо потрібні пакети -- сам антивірус та його демон \textit{[1, 2]}.
	
	\begin{minted}{bash}
		sudo apt install clamav clamav-daemon
	\end{minted}
	
	Далі потрібно вирішити питання з правами доступу. Коли Exim отримує лист, він розпаковує його у свою тимчасову директорію для сканування. Щоб антивірус міг прочитати ці розпаковані файли, нам потрібно додати користувача \texttt{clamav} до групи поштового сервера \textit{[1, 2]}.
	
	\begin{minted}{bash}
		sudo usermod -aG Debian-exim clamav
	\end{minted}
	
	Тепер навчимо Exim взаємодіяти з антивірусом. Відкриваємо файл з макросами -- \texttt{/etc/exim4/conf.d/main/01\_exim4-config\_listmacrosdefs}. Додаємо рядок, вказаний нижче. Це вкаже поштовому серверу точний шлях до системного сокета, через який спілкується служба ClamAV \textit{[1, 2]}.
	
	\begin{minted}{bash}
		# System socket for ClamAV
		av_scanner = clamd:/var/run/clamav/clamd.ctl
	\end{minted}
	
	Після цього потрібно увімкнути саме правило блокування заражених листів. Відкриваємо файл перевірки даних \texttt{/etc/exim4/conf.d/acl/40\_exim4-config\_check\_data}. Знаходимо закоментовані блоки, які відповідають за перевірку на шкідливе програмне забезпечення. Розкоментовуємо їх. Також, рядок \texttt{spam = nobody:true} замінюємо на \texttt{spam = debian-spamd:true} \textit{[1, 2]}.
	
	\begin{minted}{bash}
		
		### acl/40_exim4-config_check_data
		#################################
		
		# This ACL is used after the contents of a message have been received. This
		# is the ACL in which you can test a message's headers or body, and in
		# particular, this is where you can invoke external virus or spam scanners.
		
		acl_check_data:
		
		# Deny if the message contains an overlong line.  Per the standards
		# we should never receive one such via SMTP.
		#
		.ifndef IGNORE_SMTP_LINE_LENGTH_LIMIT
		deny
		condition  = ${if > {$max_received_linelength}{998}}
		message    = maximum allowed line length is 998 octets, \
		got $max_received_linelength
		.endif
		
		# Deny if the headers contain badly-formed addresses.
		#
		.ifndef NO_CHECK_DATA_VERIFY_HEADER_SYNTAX
		deny
		!acl = acl_local_deny_exceptions
		!verify = header_syntax
		message = header syntax
		log_message = header syntax ($acl_verify_message)
		.endif
		
		
		# require that there is a verifiable sender address in at least
		# one of the "Sender:", "Reply-To:", or "From:" header lines.
		.ifdef CHECK_DATA_VERIFY_HEADER_SENDER
		deny
		!acl = acl_local_deny_exceptions
		!verify = header_sender
		message = No verifiable sender address in message headers
		.endif
		
		
		# Deny if the message contains malware. Before enabling this check, you
		# must install a virus scanner and set the av_scanner option in the
		# main configuration.
		#
		# exim4-daemon-heavy must be used for this section to work.
		#
		deny
		malware = *
		message = This message was detected as possible malware ($malware_name).
		
		
		# Add headers to a message if it is judged to be spam. Before enabling this,
		# you must install SpamAssassin. You may also need to set the spamd_address
		# option in the main configuration.
		#
		# exim4-daemon-heavy must be used for this section to work.
		#
		# Please note that this is only suiteable as an example. See
		# /usr/share/doc/exim4-base/README.Debian.gz
		#
		# See the exim docs and the exim wiki for more suitable examples.
		#
		# # Remove internal headers
		# warn
		#   remove_header = X-Spam_score: X-Spam_score_int : X-Spam_bar : \
		#                   X-Spam_report
		#
		warn
		condition = ${if <{$message_size}{120k}{1}{0}}
		#   # ":true" to add headers/acl variables even if not spam
		spam = debian-spamd:true
		add_header = X-Spam_score: $spam_score
		add_header = X-Spam_bar: $spam_bar
		#   # Do not enable this unless you have shorted SpamAssassin's report
		#   #add_header = X-Spam_report: $spam_report
		#
		# Reject spam messages (score >15.0).
		# This breaks mailing list and forward messages.
		deny
		condition = ${if <{$message_size}{120k}{1}{0}}
		condition = ${if >{$spam_score_int}{150}{true}{false}}
		message = Classified as spam (score $spam_score)
		
		
		# This hook allows you to hook in your own ACLs without having to
		# modify this file. If you do it like we suggest, you'll end up with
		# a small performance penalty since there is an additional file being
		# accessed. This doesn't happen if you leave the macro unset.
		.ifdef CHECK_DATA_LOCAL_ACL_FILE
		.include CHECK_DATA_LOCAL_ACL_FILE
		.endif
		
		# accept otherwise
		accept
	\end{minted}

	\subsection*{SpamAssassin}
	
	Як зазвичай, починаємо зі встановлення пакету \textit{[1, 2]}.
	
	\begin{minted}{bash}
		sudo apt install spamassassin
	\end{minted}
	
	Тепер потрібно вказати поштовому серверу Exim, як зв'язатися з цим фільтром. Відкриваємо файл \texttt{/etc/exim4/conf.d/main/01\_exim4-config\_listmacrosdefs} та додаємо рядки \textit{[1, 2]}:
	
	\begin{minted}{bash}
		# Define SpamAssassin address and port
		spamd_address = 127.0.0.1 783
	\end{minted}
	
	Далі треба розкоментувати нижчевказані рядки в файлі \texttt{/etc/exim4/conf.d/acl/40\_exim4-config\_check\_data}. Це вже було зроблено раніше, але варто перевірити \textit{[1, 2]}.
	
	\begin{minted}{bash}
		# Uncomment these lines to reject spam messages
		deny
			condition = ${if <{$message_size}{120k}{1}{0}}
			condition = ${if >{$spam_score_int}{150}{true}{false}}
			message = Classified as spam (score $spam_score)
	\end{minted}
	
	Зберігаємо файл і запускаємо службу SpamAssassin, а також перезапускаємо Exim, щоб він підхопив нові конфігурації \textit{[1, 2]}.
	
	\begin{minted}{bash}
		# Enable and start SpamAssassin service
		sudo systemctl enable --now spamd
		
		# Restart spamd
		sudo systemctl restart spamd
		
		# Restart Exim to apply new ACL and macros
		sudo systemctl restart exim4
	\end{minted}
	
	Окрім цього, оновимо базу даних правил спаму \textit{[1, 2]}.
	
	\begin{minted}{bash}
		# Download and compile the latest SpamAssassin rule updates
		sudo sa-update
		
		# Restart the daemon to load the new rules into memory
		sudo systemctl restart spamd
	\end{minted}
	
	Для перевірки з локалхосту, додамо кастомні правила \textit{[1, 2]}.
	
	\begin{minted}{bash}
		# Add a custom rule to detect the GTUBE string and assign it a massive score
		echo "body LOCAL_GTUBE /GTUBE-STANDARD-ANTI-SPAM-TEST-EMAIL/" | sudo tee -a /etc/spamassassin/local.cf
		echo "score LOCAL_GTUBE 200.0" | sudo tee -a /etc/spamassassin/local.cf
		
		# Restart the service to load our new custom rule
		sudo systemctl restart spamd
	\end{minted}
	
	Для перевірки спам-фільтрів існує спеціальний безпечний рядок GTUBE, який працює за тим самим принципом, що й EICAR для вірусів. Спам-фільтри в усьому світі розпізнають його і миттєво нараховують листу 1000 балів спаму. Перевіряємо роботу фільтру.
	
	\begin{minted}{bash}

		Password: rootpass
		
		=== Trying mail.cat.com:25...
	\end{minted}

	\subsection*{Висновок}
	
	Під час виконання лабораторної роботи було успішно розгорнуто та налаштовано повноцінну інфраструктуру електронної пошти на базі двох віртуальних машин. Було налаштовано сервери передачі повідомлень Exim та агенти доставки Dovecot. Для забезпечення конфіденційності передачі даних створено локальний центр сертифікації та налаштовано шифрування за протоколом TLS. Було впроваджено сучасні механізми захисту від підробки відправника, такі як SPF та криптографічні підписи DKIM, які успішно пройшли перевірку. Крім того, налаштовано системи контентної фільтрації, зокрема антивірус ClamAV та спам-фільтр SpamAssassin, що гарантує захист від шкідливого програмного забезпечення та небажаної пошти. Працездатність усіх компонентів, включно з протоколами SMTP, IMAPS та POP3S, була успішно підтверджена за допомогою спеціалізованих мережевих утиліт.
	
	\section*{Контрольні запитання.}
	
	\begin{enumerate}
		\item \textbf{Архітектура електронної пошти.} \\ Базова архітектура електронної пошти складається з клієнтської та серверної частин. Користувач взаємодіє з поштовим клієнтом (MUA -- Mail User Agent), який відправляє листи на поштовий сервер (MTA -- Mail Transfer Agent). MTA відповідає за маршрутизацію та передачу листа через інтернет до сервера отримувача. На стороні отримувача агент доставки (MDA -- Mail Delivery Agent) приймає лист і зберігає його в поштовій скриньці, звідки користувач може його завантажити або прочитати за допомогою свого клієнта (MRA -- Mail Retrieval Agent).
		
		\item \textbf{Призначення та принципи роботи протоколів SMTP, POP3 та IMAP.} \\ SMTP (Simple Mail Transfer Protocol) використовується виключно для відправлення листів від клієнта до сервера та для пересилки між самими серверами. POP3 (Post Office Protocol version 3) призначений для завантаження вхідних листів із сервера на локальний комп'ютер, при цьому листи зазвичай видаляються з сервера після завантаження. IMAP (Internet Message Access Protocol) дозволяє працювати з листами безпосередньо на сервері, синхронізуючи папки, статуси прочитання та зберігаючи всю пошту на сервері для доступу з різних пристроїв.
		
		\item \textbf{Формат поштового повідомлення.} \\ Поштове повідомлення складається з трьох основних частин: конверта, заголовків та тіла. Конверт містить інформацію про відправника та отримувача, яка використовується протоколом SMTP для маршрутизації між серверами. Заголовки містять метадані, такі як тема, дата, технічна інформація про маршрут, версія поштового клієнта та криптографічні підписи. Тіло містить безпосередньо текст листа та прикріплені файли, які зазвичай кодуються за стандартом MIME.
		
		\item \textbf{Маршрутизації повідомлень електронної пошти.} \\ Маршрутизація здійснюється переважно за допомогою системи доменних імен (DNS). Коли MTA потрібно відправити лист, він робить запит до DNS на отримання MX-записів (Mail Exchanger) для домену отримувача. Цей запис вказує на доменне ім'я та пріоритет поштового сервера, який обслуговує цей домен. Потім MTA дізнається IP-адресу цього сервера і встановлює з ним SMTP-з'єднання для передачі повідомлення.
		
		\item \textbf{Методи автентифікації поштових повідомлень DKIM та SPF.} \\ SPF (Sender Policy Framework) -- це TXT-запис у системі DNS, який містить точний список IP-адрес або серверів, яким дозволено відправляти пошту від імені цього домену. DKIM (DomainKeys Identified Mail) -- це метод, за якого сервер-відправник додає до листа криптографічний підпис, згенерований за допомогою закритого ключа. Сервер-отримувач перевіряє цей підпис за допомогою відкритого ключа, розміщеного в DNS відправника, що гарантує цілісність листа та підтверджує домен відправника.
		
		\item \textbf{Технологія DMARC.} \\ DMARC (Domain-based Message Authentication, Reporting, and Conformance) -- це політика, яка логічно об'єднує механізми SPF та DKIM. Вона вказує серверу-отримувачу, що саме робити з листами, які не пройшли перевірку автентичності (наприклад, пропустити, помістити в папку спаму або повністю відхилити). Крім того, DMARC дозволяє власникам доменів отримувати детальні звіти про результати перевірки їхніх листів на інших серверах.
	\end{enumerate}
